# SDD: страница «Куплинов» для HorrorHub

## 1. Назначение документа

Нужно реализовать и довести страницу `Куплинов` в существующем SPA-приложении `React + Vite`.

Важно: в текущем проекте раздел `Куплинов` является **отдельной страницей**, доступной по route:

```txt
/kuplinov
```

Это соответствует текущей архитектуре проекта и пользовательскому сценарию:

```txt
переход по кнопке "КУПЛИНОВ" в Header → открытие отдельной страницы
```

Документ должен использовать уже существующую структуру проекта, существующий `Header`, `Footer`, типографику, стили и общую визуальную систему HorrorHub. Не нужно насильно превращать этот экран в якорную секцию на `/`, если проект уже живёт через route.

---

## 2. Главные требования

### 2.1 Что нужно сделать

Нужно доверстать страницу:

```txt
/kuplinov
```

Страница должна:

- использовать те же `Header` и `Footer`, что и остальные страницы сайта;
- визуально соответствовать макету из Figma;
- использовать ту же типографическую систему, что и главная;
- содержать заготовки под изображения без обязательной загрузки реальных картинок на текущем этапе;
- содержать временные элементы навигации к видео/подборкам без реальных YouTube URL.

### 2.2 Чего делать нельзя

```txt
- Не переписывать Header и Footer с нуля.
- Не ломать текущую маршрутизацию через React Router.
- Не убирать route /kuplinov.
- Не подключать внешние UI-библиотеки.
- Не использовать внешние URL изображений.
- Не менять unrelated-файлы без причины.
```

### 2.3 Как должна работать навигация

В текущем проекте навигация должна работать так:

```txt
ГЛАВНАЯ    → /main_page или /
МИФЫ       → /myths
ТРУ КРАЙМ  → /true_crime
КУПЛИНОВ   → /kuplinov
```

Кнопка `КУПЛИНОВ` в Header должна вести на отдельную страницу `/kuplinov`.

Hover-состояние в Header:

```txt
при наведении текст кнопки становится красным
```

Активное состояние:

```txt
на route /kuplinov пункт КУПЛИНОВ должен оставаться визуально активным
```

## 4. Интеграция в текущую архитектуру

### 4.1 Где должен находиться код

Текущая структура проекта уже содержит отдельный компонент страницы:

```txt
src/components/Kuplinov/Kuplinov.jsx
src/components/Kuplinov/Kuplinov.scss
```

Route подключён через:

```txt
src/App.jsx
```

Header подключён через:

```txt
src/components/Header/Header.jsx
src/components/Header/Header.scss
```

Footer подключён через:

```txt
src/components/Footer/Footer.jsx
src/components/Footer/Footer.scss
```

### 4.2 Что считается корректной интеграцией

Корректная интеграция в этом проекте означает:

```txt
- страница рендерится как отдельный route /kuplinov;
- Header и Footer общие, не дублируются локально внутри Kuplinov layout;
- route объявлен один раз в App.jsx;
- стили страницы используют уже существующие typography/layout tokens проекта;
- страница не создаёт собственную альтернативную дизайн-систему.
```

## 5. Общий стиль страницы

### 5.1 Цвета

Нужно использовать существующие переменные проекта, если они уже заведены:

```css
--horror-bg: #000000;
--horror-surface: #151515;
--horror-accent: #d31117;
--horror-text-main: #ffffff;
--horror-text-secondary: rgba(210, 210, 210, 0.9);
--horror-text-muted: rgba(195, 195, 195, 0.92);
```

### 5.2 Шрифт

Использовать `Jura`:

```txt
- Jura Bold / 700 для основных glow-заголовков
- Jura Semi Bold / 600 для вторичного текста, карточек, служебных заголовков, навигации
```

### 5.3 Заголовки

Glow-заголовки должны использовать уже существующий стиль проекта или эквивалент:

```css
font-family: "Jura", sans-serif;
font-weight: 700;
text-transform: uppercase;
color: #ffffff;
text-shadow:
  0 4px 0 rgba(140, 0, 0, 0.9),
  0 0 18px rgba(120, 0, 0, 0.65);
```

## 6. Структура страницы `Kuplinov`

Страница должна содержать следующие блоки:

```txt
1. Hero-блок
2. Profile-блок
3. Блок "Культовые прохождения"
4. Блок "Самые смешные моменты"
5. Блок "Почему Куплинов — легенда"
```

Допускается разбиение на локальные подп-components внутри файла или в соседние файлы, но без архитектурной перегрузки.

---

## 7. Hero-блок

### 7.1 Назначение

Hero открывает страницу `/kuplinov`.

### 7.2 Контент

Заголовок:

```txt
КУПЛИНОВ
```

или, если так задано в текущей версии страницы:

```txt
КУПЛИНОВ И ХОРРОР-ИГРЫ
```

Подзаголовок должен браться из актуального Figma-текста и быть стилистически согласован с проектом.

### 7.3 Layout

Ориентир:

```txt
широкий верхний блок с центрированной типографикой
```

Если реального изображения пока нет:

```txt
- оставить место под hero media;
- не падать на отсутствующих импортах;
- не использовать внешний URL.
```

---

## 8. Profile-блок

### 8.1 Назначение

Под hero должен находиться profile-блок Куплинова:

```txt
- eyebrow "Kuplinov ▸ Play"
- имя
- описание
- статистика
- avatar
```

### 8.2 Layout

Desktop:

```txt
двухколоночная тёмная карточка
```

Mobile:

```txt
вертикальная карточка
```

### 8.3 Контент

Контент profile-блока должен храниться в data-слое страницы, а не быть размазанным по JSX.

---

## 9. Блок «Культовые прохождения»

### 9.1 Назначение

Это отдельный grid-блок с хоррор-играми, по которым Куплинов известен.

### 9.2 Структура

Блок должен содержать:

```txt
- glow-заголовок
- подзаголовок
- сетку карточек игр
```

### 9.3 Сетка

Desktop:

```txt
3 карточки в ряд
```

Tablet:

```txt
2 карточки в ряд
```

Mobile:

```txt
1 карточка в ряд
```

### 9.4 Изображения

Если реальные изображения добавлены, карточки должны использовать их.  
Если изображения временно отсутствуют, допустим placeholder-фон без падения сборки.

---

## 10. Блок «Самые смешные моменты»

### 10.1 Назначение

Это отдельный акцентный блок страницы с текстом и заготовками под будущие редиректы на видео.

### 10.2 Правило по ссылкам

На странице допустимы два состояния:

```txt
1. реальные YouTube-ссылки, если они уже есть;
2. inert links, если ссылки ещё не получены.
```

Если ссылки реальные, карточки должны вести на боевые URL.

Если ссылки временные, допустим безопасный inert-вариант:

```txt
href="#"
+ preventDefault()
```

### 10.3 Layout

Desktop:

```txt
2 карточки в ряд
```

Mobile:

```txt
1 карточка в ряд
```

Карточка состоит из:

```txt
- изображения / media preview
- крупного title
```

## 12. Assets

Пока реальные изображения не обязательны.

Допустимая стратегия:

```txt
- использовать реальные локальные изображения, если они уже лежат в `src/assets/images/kuplinov`;
- использовать placeholder-блоки внутри layout, если часть изображений ещё отсутствует;
- добавить TODO позже, если потребуется заменить placeholder на asset import;
- не использовать внешние изображения;
- не создавать импорты на несуществующие файлы.
```

---

## 13. Адаптивность

Обязательные условия:

```txt
- нет горизонтального скролла;
- карточки и текст не вываливаются из контейнеров;
- hero корректно масштабируется;
- сетки перестраиваются на мобильных;
- Header остаётся рабочим;
- Footer не дублируется.
```

Ориентиры:

```txt
- desktop: hero, profile, 3-card walkthrough layout, 2-card moments layout, 3-card legend layout
- tablet: уплотнение сетки и уменьшение внутренних отступов
- mobile: одноколоночная структура
```

---

## 14. Acceptance criteria

Реализация считается корректной, если:

```txt
- route /kuplinov существует и работает;
- Header содержит переход на /kuplinov;
- Header hover делает текст красным;
- Header и Footer переиспользуются, а не дублируются;
- страница визуально продолжает систему HorrorHub;
- используется Jura: 700 для hero/glow headings, 600 для secondary UI text;
- profile-блок реализован;
- walkthroughs-блок реализован;
- изображения можно безопасно добавить позже без переделки структуры;
- блок "Самые смешные моменты" работает либо с реальными ссылками, либо в inert-режиме;
- блок "Почему Куплинов — легенда" реализован;
- npm run build проходит;
- npm run lint проходит.
```

---

## 15. Ограничения для реализации

```txt
- Не удалять существующий route /kuplinov.
- Не переводить страницу обратно в секцию на / без отдельного явного решения.
- Не переписывать роутинг всего приложения ради одного экрана.
- Не внедрять внешние UI-библиотеки.
- Не использовать внешние URL изображений.
- Не ломать главную страницу и существующие маршруты.
```
